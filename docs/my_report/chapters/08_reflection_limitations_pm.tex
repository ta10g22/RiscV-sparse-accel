\chapter{Reflection}
\label{ch:reflection}

\section{What Worked Well}

Structuring development as a sequence of discrete, measurable optimisation
stages (baseline, DSP MAC, and INT8 packing) was the single most valuable
decision in the project. A working baseline was established and verified on
hardware before any optimisation was attempted, and each subsequent stage
changed exactly one aspect of the design: arithmetic parallelism first, then
memory traffic. This made it possible to attribute each measured improvement
to a specific architectural change, to detect regressions immediately, and to
build the evidence-based narrative used throughout Chapter~\ref{ch:results}.

The deterministic benchmark harness also helped because fixed seeds and
on-board golden checks made regressions immediately visible after each
rebuild.

The roofline analysis in Chapter~\ref{ch:motivation} was useful because it
prevented the project from over-focusing on wider MAC arrays; the later INT8
result confirmed that memory traffic was the higher-leverage bottleneck.

\section{What Went Wrong}

The most time-consuming debugging issue was a stale \texttt{firmware.mif}
file. The board kept running an old firmware image even after the C code had
been rebuilt, which made correct RTL look faulty. The lesson was that software
and hardware artefacts must be tied together explicitly in the build system; a
Makefile dependency on \texttt{firmware.mif} would have prevented this.

Timing closure also became harder as parallel logic was added. The INT8
datapath increased routing pressure around the C-tile and MAC buses, and
Quartus only closed timing after switching to performance-oriented fitting.
This showed that FPGA parallelism is limited not only by ALMs and DSPs, but
also by placement and routing.

\section{Project Management}

The implementation and integration phases overran their planned allocation,
mainly because the originally specified interrupt-driven completion path
from the accelerator to the CPU proved too complex to integrate with the
PicoRV32 exception model and was descoped in favour of a simple polling loop
on the status register. In Semester~2, benchmarking was delayed when the
UART peripheral bundled with the PicoRV32 reference design turned out not to
support the cycle-accurate timing readouts required by the measurement
methodology in Chapter~\ref{ch:methodology}; adopting the open-source
\texttt{simpleuart} core resolved this. Preserving a fully working,
deployable design at the end of every stage, rather than accumulating
partially integrated features, was the practice that kept the overall
schedule recoverable despite these setbacks.

The project was managed through Git from the first commit, and the
repository was treated as the single source of truth for both code and
design artefacts. Over the eight months of development the repository
accumulated 73 commits spanning every layer of the system: RTL,
simulation testbenches, firmware, Quartus project files, and the
report itself. Committing after every measurable change preserved the
staged optimisation history as a linear, inspectable timeline. The
baseline accelerator, the DSP-mapped MAC enhancement, and the INT8
packed mode each appear as identifiable commit clusters that can be
checked out and rebuilt independently, which is also what makes the
stage-to-stage cycle reductions in Chapter~\ref{ch:results}
reproducible. The GitHub remote additionally guarded against local
hardware loss, which is non-trivial for an FPGA project whose build
artefacts span Quartus output, firmware ELF files, generated MIF
files, and synthesis reports. The repository structure is documented
in Appendix~\ref{app:archive}.

Reading notes and design artefacts were kept in topic-organised
digital folders on an iPad throughout the project. Separate folders
held learned concepts (CSR encoding, symmetric quantisation, roofline
reasoning), initial planning material, top-level SoC block diagrams,
and FSM chart drafts. Keeping these artefacts in a structured
collection rather than scattered loose files meant that the freehand
designs that ultimately fed Figures~\ref{fig:soc-block}
and~\ref{fig:accel-fsm} remained traceable to their final realisations,
and that any concept revisited during implementation could be located
within seconds rather than re-derived from scratch.

The accelerator was deliberately broken into smaller, independently
testable blocks rather than approached as a single integration effort.
The control FSM, the parallel MAC datapath, the \texttt{C\_Tile}
register file, and the MMIO front-end were each implemented and
unit-tested in isolation before being wired into \texttt{accel\_top}.
This block-first approach made it possible to make rapid progress on
each component, contain debugging within a single module hierarchy at
a time, and only bring the integration testbench into play once each
block had passed its own directed and constrained-random checks.

For the same reason, full accelerator-level verification was completed
before the accelerator was integrated into the SoC. The four-tier
simulation strategy described in Chapter~\ref{ch:methodology}
(datapath unit, control-FSM unit, accelerator-top, integration) was
deliberately ordered so that the accelerator was already known-correct
in isolation before it had to coexist with PicoRV32, the shared BRAM,
and the firmware. This drastically reduced the failure space when
bugs surfaced after FPGA programming: a regression in firmware-driven
runs could immediately be attributed to the integration layer
(memory map, MMIO sequencing, firmware) rather than the accelerator's
internal logic, because that logic had been independently certified
against a software scoreboard.

The overarching lesson is that hardware project timelines are dominated by
integration and debugging rather than initial design. The Gantt charts in
Appendix~\ref{app:gantt} record both the original plan and the way the work
actually unfolded.
